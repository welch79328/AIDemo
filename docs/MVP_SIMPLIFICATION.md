# MVP 簡化設計說明

**更新日期**: 2026-03-20
**變更原因**: 加速 MVP 開發，專注核心功能展示

---

## 🎯 簡化原則

### MVP 版本目標
> **快速展示業務行動成效評估的核心價值**，而非完整的生產系統。

重點在於：
1. ✅ 證明「客戶健檢 30 問」數位化的可行性
2. ✅ 展示智能客戶分類（AA 客戶判定、成熟度評分）
3. ✅ 展示業務成效追蹤（轉換漏斗、儀表板）
4. ✅ 收集用戶反饋，驗證需求

---

## 🔄 主要簡化項目

### 1. ❌ 移除登入認證系統

#### 原設計
- JWT Token 認證
- 用戶註冊/登入
- 權限控制（業務/主管/管理員）
- Session 管理

#### 簡化後
- **無需登入**，直接進入系統
- 所有功能開放存取
- 暫時不區分用戶身份

#### 理由
- **加速開發**：省去 2-3 天認證系統開發時間
- **簡化測試**：無需每次測試都登入
- **專注核心**：優先展示業務邏輯，而非技術細節
- **易於演示**：開啟瀏覽器即可使用，無需帳號設定

#### 未來擴展
- Phase 2 可快速加回認證系統
- 使用者模型已設計完成，隨時可啟用

---

### 2. 📊 簡化資料模型

#### User 表的處理
- **保留** User 表結構（已在 database models 中定義）
- **暫不實作** 註冊/登入功能
- **未來擴展** 時可快速啟用

#### 簡化後的核心資料表
```
核心業務表（MVP 重點）:
├── customers          ✅ 客戶資料
├── visits             ✅ 拜訪記錄（一訪/二訪）
├── contracts          ✅ 簽約與導入 KPI
└── performance_metrics ✅ 業務績效

支援表（保留但暫不使用）:
├── users              📁 保留結構，暫不實作
├── question_templates 📁 可選，硬編碼問題即可
└── aa_customer_criteria 📁 可選，判定邏輯寫死即可
```

## 🔐 安全性考量

- **僅限內部演示**：不對外開放

## 🎬 使用場景

### MVP 演示流程

#### 情境 1: 業務拜訪客戶後記錄
```
1. 開啟瀏覽器 → http://localhost:5173
2. 點擊「客戶管理」
3. 點擊「新增客戶」，填寫基本資料
4. 儲存客戶資料
5. 點擊「新增拜訪」，填寫一訪問卷
6. 系統自動計算成熟度評分
7. 系統自動判定是否為 AA 客戶
8. 查看儀表板，看到統計更新
```

#### 情境 2: 主管查看團隊成效
```
1. 開啟瀏覽器 → http://localhost:5173
2. 點擊「儀表板」
3. 查看本月拜訪次數、簽約數量
4. 查看轉換漏斗圖表
5. 查看 AA 客戶列表
6. 點擊客戶查看詳細拜訪記錄
```

### 無需登入的便利性
- ✅ **快速展示**：打開即用，無需帳號設定
- ✅ **團隊協作**：多人同時使用同一資料庫
- ✅ **簡化培訓**：無需教導登入流程

## 💬 常見問題

### Q1: 沒有登入，資料不會亂掉嗎？
**A**: MVP 階段僅供內部演示，使用同一個資料庫。正式上線前會加回認證系統。

### Q2: 多人同時使用會衝突嗎？
**A**: 不會。每筆資料有獨立 ID，不同人新增的客戶不會衝突。反而更方便團隊協作測試。

### Q3: 未來加回登入會很麻煩嗎？
**A**: 不會。資料模型已設計完成，只需 2-3 天即可加回。

### Q4: 簡化後還能展示完整功能嗎？
**A**: 可以。核心業務邏輯完全保留，只是簡化了用戶管理流程。
