# sales-lead-management - 需求規格

## 專案概述

### 背景
JGB Smart Property (金箍棒智慧物管) 需要一個整合 AI 的銷售潛在客戶管理系統，用於管理從臉書廣告獲取的潛在客戶名單，追蹤銷售互動過程，並透過 AI 分析對話內容進行客戶健檢評估，以識別高價值的 AA 級客戶。

### 目標
1. 自動化潛在客戶名單的導入和管理流程
2. 完整記錄銷售人員與潛在客戶的互動過程
3. 運用 AI 技術分析銷售對話，評估客戶品質
4. 標準化客戶健檢流程，提升銷售效率
5. 產生結構化的健檢報告，支援後續決策

### 範圍
**包含的功能**:
- 從 Excel 檔案導入臉書廣告名單
- 潛在客戶資料管理和查詢
- 銷售互動記錄（文字文檔、錄音檔）
- AI 對話分析和業務30問匹配
- 客戶分級評估（AA 客戶識別）
- 健檢紀錄表生成和匯出

**不包含的功能**:
- 臉書廣告投放和管理
- 自動撥號或聯絡功能
- 錄音設備整合
- 客戶關係管理（CRM）的完整功能

## 需求詳細說明

### 需求 1: 資料導入管理

#### 1.1 Excel 名單導入
- 當使用者上傳 Excel 檔案時，系統應解析檔案內容並提取潛在客戶資料
- 系統應支援導入「20220714_臉書廣告表單_20250502更新.xlsx」格式的檔案
- 當檔案格式不符時，系統應顯示明確的錯誤訊息並說明正確格式

**驗收標準**:
- [ ] 系統能成功解析包含「3特點輪播廣告」欄位的 Excel 檔案
- [ ] 系統能提取客戶姓名、聯絡方式、廣告來源等必要欄位
- [ ] 當檔案格式錯誤時，系統顯示具體的錯誤訊息
- [ ] 系統能處理包含至少 1000 筆資料的 Excel 檔案

#### 1.2 資料驗證與去重
- 當導入名單時，系統應驗證必填欄位的完整性
- 當發現重複的聯絡資料時，系統應提示使用者並提供合併或保留選項
- 系統應記錄每筆資料的導入時間和來源檔案

**驗收標準**:
- [ ] 系統能識別並標示缺少必填欄位的資料列
- [ ] 系統能檢測重複的電話號碼或電子郵件
- [ ] 每筆導入的資料都包含時間戳記和來源檔案名稱
- [ ] 使用者能選擇覆蓋或保留重複資料

#### 1.3 導入歷史記錄
- 系統應記錄每次資料導入的詳細資訊（時間、使用者、檔案名稱、筆數）
- 使用者應能查看歷史導入記錄和統計資訊

**驗收標準**:
- [ ] 系統保存所有導入操作的完整記錄
- [ ] 使用者能查看導入成功/失敗的筆數統計
- [ ] 使用者能下載導入錯誤報告

### 需求 2: 潛在客戶管理

#### 2.1 客戶資料檢視
- 系統應以列表形式顯示所有潛在客戶資料
- 使用者應能按照姓名、電話、匯入日期、聯絡狀態等條件搜尋和篩選客戶
- 系統應顯示每位客戶的基本資料和聯絡狀態

**驗收標準**:
- [ ] 客戶列表包含姓名、電話、來源、狀態等關鍵欄位
- [ ] 搜尋功能能在 1 秒內返回結果
- [ ] 使用者能同時應用多個篩選條件
- [ ] 列表支援分頁，每頁顯示 20-50 筆資料

#### 2.2 客戶資料編輯
- 使用者應能編輯客戶的基本資料
- 當客戶資料被修改時，系統應記錄修改歷史
- 系統應防止多人同時編輯同一客戶資料造成的衝突

**驗收標準**:
- [ ] 使用者能更新客戶的姓名、電話、地址等欄位
- [ ] 系統保存完整的修改歷史（修改時間、修改人、修改內容）
- [ ] 當資料被他人編輯時，系統顯示警告訊息

#### 2.3 聯絡狀態管理
- 系統應追蹤每位客戶的聯絡狀態（未聯絡、已聯絡、待跟進、已成交、已放棄）
- 當使用者更新聯絡狀態時，系統應記錄狀態變更時間
- 系統應支援依聯絡狀態產生統計報表

**驗收標準**:
- [ ] 系統提供至少 5 種聯絡狀態供選擇
- [ ] 狀態變更時自動記錄時間戳記
- [ ] 使用者能查看各狀態的客戶數量統計
- [ ] 系統能產生聯絡狀態分布圖表

### 需求 3: 銷售互動記錄

#### 3.1 互動文檔上傳
- 當使用者與客戶互動後，系統應允許上傳相關文檔（Word、PDF、圖片等）
- 系統應將文檔與對應的客戶資料關聯
- 系統應記錄文檔上傳時間和上傳者

**驗收標準**:
- [ ] 系統支援上傳 .doc, .docx, .pdf, .jpg, .png 格式的檔案
- [ ] 單一檔案大小限制為 10MB
- [ ] 每位客戶能關聯多個文檔檔案
- [ ] 使用者能預覽和下載已上傳的文檔

#### 3.2 錄音檔上傳與管理
- 當使用者上傳銷售通話錄音時，系統應支援常見音訊格式（MP3、WAV、M4A）
- 系統應顯示錄音檔的基本資訊（檔名、時長、上傳時間）
- 系統應提供線上播放功能

**驗收標準**:
- [ ] 系統支援 .mp3, .wav, .m4a 格式的音訊檔案
- [ ] 單一錄音檔大小限制為 50MB
- [ ] 系統能顯示錄音檔的時長資訊
- [ ] 使用者能在瀏覽器中直接播放錄音

#### 3.3 互動記錄時間軸
- 系統應以時間軸形式顯示與客戶的所有互動記錄
- 時間軸應包含文檔上傳、錄音上傳、狀態變更等所有活動
- 使用者應能快速查看最近的互動內容

**驗收標準**:
- [ ] 時間軸按時間倒序顯示所有互動記錄
- [ ] 每筆記錄顯示時間、操作類型、操作人
- [ ] 使用者能直接從時間軸存取相關文檔或錄音
- [ ] 時間軸支援篩選特定類型的互動記錄

### 需求 4: AI 對話分析

#### 4.1 音訊轉文字
- 當使用者上傳錄音檔後，系統應提供將錄音轉換為文字稿的功能
- 系統應支援繁體中文語音識別
- 轉換完成後，系統應通知使用者並顯示文字稿

**驗收標準**:
- [ ] 系統能將繁體中文錄音轉換為文字
- [ ] 轉換準確率達到 85% 以上
- [ ] 30 分鐘的錄音應在 10 分鐘內完成轉換
- [ ] 使用者能編輯和修正自動生成的文字稿

#### 4.2 業務30問匹配分析
- 當文字稿或文檔可用時，系統應使用 AI 分析內容並匹配「業務30問」題庫
- 系統應識別對話中涵蓋的業務問題，並提取客戶的回答
- 系統應標示哪些問題已被討論、哪些問題尚未涉及

**驗收標準**:
- [ ] 系統能載入「5.00.8.3_導入顧問服務_業務30問_客戶健檢表」作為題庫
- [ ] AI 能識別對話中對應的業務問題（識別率 ≥ 80%）
- [ ] 系統能提取並結構化客戶的回答內容
- [ ] 系統能生成「已討論」與「未討論」問題的清單

#### 4.3 對話品質評估
- 系統應分析銷售對話的完整性和深度
- 系統應評估業務人員是否充分介紹 JGB Smart Property 的服務內容
- 系統應提供對話改進建議

**驗收標準**:
- [ ] 系統能計算業務30問的覆蓋率（已討論問題數/總問題數）
- [ ] 系統能識別對話中提及的 JGB Smart Property 服務項目
- [ ] 系統能標示缺失的關鍵銷售要點
- [ ] 系統能產生對話品質評分（1-100 分）

### 需求 5: 客戶分級評估

#### 5.1 AA 客戶識別
- 當 AI 分析完成後，系統應根據客戶回答評估客戶等級
- 系統應識別符合 AA 客戶標準的潛在客戶
- 系統應清楚標示 AA 客戶，方便業務團隊優先跟進

**驗收標準**:
- [ ] 系統能根據業務30問的回答計算客戶評分
- [ ] 系統提供明確的 AA 客戶判定標準（可配置）
- [ ] AA 客戶在列表中有明顯的視覺標示
- [ ] 使用者能查看 AA 客戶的判定依據和詳細評分

#### 5.2 客戶分級規則配置
- 管理員應能配置客戶分級的評分規則
- 系統應支援根據不同業務問題設定不同權重
- 當規則變更時，系統應允許重新評估現有客戶

**驗收標準**:
- [ ] 管理員能設定 AA、A、B、C 等級的評分門檻
- [ ] 管理員能調整每個業務問題的權重（1-10）
- [ ] 規則變更後，使用者能觸發批次重新評估
- [ ] 系統保存歷史評分規則的版本記錄

#### 5.3 評估結果追蹤
- 系統應記錄每次客戶評估的結果和時間
- 當客戶等級發生變化時，系統應記錄變更歷史
- 使用者應能查看客戶等級的變化趨勢

**驗收標準**:
- [ ] 系統保存每次評估的完整記錄（時間、評分、等級）
- [ ] 使用者能查看客戶等級的歷史變更
- [ ] 系統能產生客戶等級分布統計圖表
- [ ] 系統能追蹤 AA 客戶的轉換率

### 需求 6: 健檢報告生成

#### 6.1 客戶健檢紀錄表生成
- 當客戶評估完成後，系統應能產生「客戶30問健檢紀錄表」
- 報告應包含所有業務30問及客戶的對應回答
- 報告應標示客戶等級和評分詳情

**驗收標準**:
- [ ] 系統能生成符合「客戶30問健檢紀錄表 (回覆).xlsx」格式的報告
- [ ] 報告包含客戶基本資料、30問回答、評分、等級
- [ ] 報告能清楚標示已討論與未討論的問題
- [ ] 報告包含生成時間和生成者資訊

#### 6.2 報告匯出功能
- 使用者應能將健檢報告匯出為 Excel 格式
- 系統應支援單一客戶報告和批次匯出多個客戶報告
- 匯出的檔案應符合既定的格式規範

**驗收標準**:
- [ ] 使用者能匯出單一客戶的健檢報告（.xlsx 格式）
- [ ] 使用者能批次匯出多個客戶報告（ZIP 壓縮檔）
- [ ] 匯出的 Excel 檔案格式與範本一致
- [ ] 檔案命名規則為「客戶姓名_健檢報告_日期.xlsx」

#### 6.3 報告檢視與列印
- 使用者應能在系統中預覽健檢報告
- 系統應提供友善列印的報告格式
- 使用者應能透過 email 分享報告

**驗收標準**:
- [ ] 系統提供報告的網頁預覽功能
- [ ] 報告格式適合 A4 紙張列印
- [ ] 使用者能直接從系統發送報告 email
- [ ] Email 包含 PDF 格式的報告附件

## 非功能性需求

### 效能需求
- 系統應在 2 秒內完成客戶列表的載入（100 筆資料）
- 系統應在 3 秒內完成單一客戶詳細資料的載入
- 音訊轉文字功能應在 10 分鐘內完成 30 分鐘的錄音轉換
- AI 分析功能應在 30 秒內完成單一對話的分析
- 系統應支援至少 50 個並發使用者同時操作

### 安全需求
- 系統應要求使用者登入才能存取客戶資料
- 系統應使用 HTTPS 加密所有網路傳輸
- 敏感的客戶資料（電話、地址）應進行加密儲存
- 系統應記錄所有使用者的操作日誌（audit log）
- 系統應實施角色權限控制（RBAC）

### 可用性需求
- 系統應達到 99% 的正常運行時間（每月停機時間不超過 7.2 小時）
- 系統應在 5 分鐘內從故障中恢復
- 系統應每日自動備份所有資料
- 系統應提供資料還原功能（最多回溯 30 天）

### 可維護性需求
- 程式碼應遵循專案定義的編碼規範
- 系統應提供詳細的操作日誌和錯誤日誌
- 系統應提供健康檢查 API 端點
- 關鍵功能應具備單元測試（測試覆蓋率 ≥ 70%）

### 使用者體驗需求
- 介面應支援繁體中文
- 系統應提供操作說明和提示訊息
- 關鍵操作（刪除、匯出）應有確認提示
- 系統應在 1 秒內對使用者操作提供視覺回饋

## 資料需求

### 資料實體

#### 1. 潛在客戶 (Lead)
- **欄位**:
  - ID（唯一識別碼）
  - 姓名（必填）
  - 電話（必填）
  - Email
  - 地址
  - 廣告來源（3特點輪播廣告）
  - 聯絡狀態
  - 客戶等級（AA/A/B/C）
  - 評分
  - 建立時間
  - 最後更新時間
  - 導入批次ID
- **關聯**: 與互動記錄、健檢報告為一對多關係

#### 2. 互動記錄 (Interaction)
- **欄位**:
  - ID
  - 客戶ID（外鍵）
  - 類型（文檔/錄音/狀態變更）
  - 檔案路徑
  - 文字稿
  - 建立時間
  - 建立者
  - 備註
- **關聯**: 屬於一位潛在客戶

#### 3. AI 分析結果 (AIAnalysis)
- **欄位**:
  - ID
  - 互動記錄ID（外鍵）
  - 業務30問回答（JSON 格式）
  - 覆蓋率
  - 品質評分
  - 分析時間
  - AI 模型版本
- **關聯**: 對應一筆互動記錄

#### 4. 客戶評估 (CustomerEvaluation)
- **欄位**:
  - ID
  - 客戶ID（外鍵）
  - 評分
  - 等級
  - 評估時間
  - 評估規則版本
  - 備註
- **關聯**: 屬於一位潛在客戶

#### 5. 健檢報告 (HealthCheckReport)
- **欄位**:
  - ID
  - 客戶ID（外鍵）
  - 報告內容（JSON）
  - 生成時間
  - 生成者
  - 檔案路徑
- **關聯**: 屬於一位潛在客戶

#### 6. 導入批次 (ImportBatch)
- **欄位**:
  - ID
  - 檔案名稱
  - 導入時間
  - 導入者
  - 成功筆數
  - 失敗筆數
  - 狀態
- **關聯**: 與潛在客戶為一對多關係

#### 7. 業務30問題庫 (BusinessQuestions)
- **欄位**:
  - ID
  - 問題編號（1-30）
  - 問題內容
  - 類別
  - 權重
  - 啟用狀態
  - 版本
- **關聯**: 獨立實體，用於 AI 匹配

### 資料驗證
- 電話號碼應符合台灣手機或市話格式
- Email 應符合標準 email 格式
- 客戶等級應限定為 AA、A、B、C 四種之一
- 評分應為 0-100 之間的整數
- 檔案類型應限定在允許的格式清單內

### 資料保留
- 客戶資料應永久保留，除非手動刪除
- 互動記錄應保留至少 3 年
- AI 分析結果應保留至少 2 年
- 操作日誌應保留至少 1 年
- 已刪除的資料應進入回收站保留 30 天後永久刪除

## 整合需求

### 外部系統整合

#### 1. AI 語音轉文字服務
- **整合方式**: RESTful API
- **資料流向**: 單向（系統上傳音訊，接收文字稿）
- **頻率**: 即時
- **需求**: 支援繁體中文語音識別

#### 2. AI 對話分析服務
- **整合方式**: RESTful API
- **資料流向**: 雙向（系統提供文字稿和題庫，接收分析結果）
- **頻率**: 即時
- **需求**: 支援自然語言理解和語意匹配

#### 3. Email 服務
- **整合方式**: SMTP 或第三方 Email API
- **資料流向**: 單向（系統發送 email）
- **頻率**: 即時
- **需求**: 支援附件發送

### API 需求
- 系統應提供 RESTful API 供其他系統整合
- API 應使用 JWT 或 API Key 進行認證
- API 應提供完整的 API 文件（Swagger/OpenAPI）
- API 回應時間應在 95% 的情況下小於 1 秒

## 使用者介面需求

### 主要畫面

#### 1. 客戶列表頁
- **功能**: 顯示所有潛在客戶，支援搜尋、篩選、排序
- **元素**: 搜尋框、篩選器、客戶表格、分頁控制、匯入按鈕

#### 2. 客戶詳細頁
- **功能**: 顯示單一客戶的完整資訊和互動歷史
- **元素**: 客戶基本資料卡、互動時間軸、文檔列表、錄音列表、健檢報告區

#### 3. 資料導入頁
- **功能**: 上傳 Excel 檔案並導入客戶名單
- **元素**: 檔案上傳區、進度顯示、導入結果摘要、錯誤報告下載

#### 4. AI 分析頁
- **功能**: 檢視 AI 分析結果和業務30問匹配情況
- **元素**: 文字稿顯示、問題匹配表、評分儀表板、改進建議

#### 5. 健檢報告頁
- **功能**: 產生和檢視客戶健檢報告
- **元素**: 報告預覽、匯出按鈕、email 分享功能、列印按鈕

#### 6. 設定頁
- **功能**: 配置客戶分級規則和業務30問題庫
- **元素**: 評分規則表單、問題清單、權重設定、儲存按鈕

### 使用者體驗
- 介面應符合現代化設計規範（Material Design 或類似）
- 常用操作應在 3 個步驟內完成
- 表單驗證應即時提供錯誤提示
- 長時間操作（導入、AI 分析）應顯示進度指示器

## 約束條件

### 技術約束
- 後端必須使用 Python（FastAPI 框架）
- 前端必須使用 Vue 3
- 資料庫必須使用 PostgreSQL
- 檔案儲存必須支援本地檔案系統或 S3 相容儲存
- 系統必須相容 Chrome、Firefox、Safari 最新版本
- 系統必須支援 RWD（響應式網頁設計）

### 業務約束
- 系統必須符合個人資料保護法（PDPA）的要求
- 客戶資料存取必須有完整的 audit trail
- 系統必須在 2026 Q2 前完成核心功能開發

### 預算約束
- AI 語音轉文字服務應考慮成本效益（可選用開源方案或商用服務）
- 雲端儲存成本應控制在合理範圍（考慮檔案壓縮和生命週期管理）
